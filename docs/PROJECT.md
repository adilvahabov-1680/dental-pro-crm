## 1. Project Overview

**Dental Pro CRM** — мульти-тенантная SaaS-платформа для стоматологов и клиник.

- Один продукт, много независимых клиник/врачей. Каждый тенант видит **только свои данные**.
- Покрывает полный цикл: пациент → приём → лечение → зубная карта → оплата → документы → напоминания.
- Полная поддержка **детской стоматологии** (молочные зубы, опекуны, возрастная логика).
- Язык интерфейса: **Azerbaijani (default)**, архитектура готова к **Russian** и **English** (i18n с первого дня).
- Целевой рынок: Азербайджан (валюта AZN, телефоны +994, SMS/WhatsApp-уведомления).

**Стек (целевой):** Next.js (App Router) + TypeScript + PostgreSQL + Prisma + Tailwind. Auth — NextAuth/Lucia с tenant-контекстом.

## 3. Роли пользователей

| Роль | Доступ |
|------|--------|
| **Super Admin** (AV Systems) | управление тенантами, планами, биллингом платформы; **не видит мед. данные** |
| **Clinic Owner / Admin** | всё внутри своей клиники: врачи, финансы, настройки, отчёты |
| **Doctor** | свои пациенты и приёмы, зубная карта, лечение; финансы — только свои показатели |
| **Reception** | календарь, запись, карточки пациентов (без мед. деталей лечения), приём оплат |
| **Accountant** *(опц.)* | финансы и отчёты, без мед. данных |
| **Patient** *(future)* | пациентский кабинет |

Соло-врач = клиника из одного пользователя с ролью Owner+Doctor (та же модель, без отдельной логики).

## 4. Multi-tenant логика

1. **Tenant = Clinic.** Таблица `clinics` — корень изоляции.
2. **Каждая бизнес-таблица содержит `clinic_id`** (patients, appointments, invoices, …). Без исключений.
3. **Изоляция на уровне запросов:** Prisma-middleware / расширение клиента автоматически добавляет `where clinic_id = session.clinicId` ко всем запросам. Опционально — PostgreSQL RLS как второй слой защиты.
4. **Пользователь принадлежит одному тенанту** (`users.clinic_id` + `role`). Super Admin — вне тенантов (`clinic_id = null`, отдельная админ-панель).
5. **Один домен**, tenant определяется после логина по сессии (subdomain-режим — future).
6. **Никаких cross-tenant ссылок** в FK; ID — UUID (не перебираемые).
7. **Файлы** хранятся с префиксом `clinic_id/` в storage; выдача только через подписанные URL после проверки тенанта.
8. **Аудит:** `audit_log (clinic_id, user_id, action, entity, before/after, timestamp)`.

## 6. MVP (v1.0)

**Цель: врач может вести пациентов, приёмы, зубную карту и деньги.**

- Auth + регистрация клиники, роли Owner/Doctor/Reception
- Multi-tenant изоляция (полностью, с первого дня — это не отложить)
- Пациенты: CRUD, поиск, детский профиль + опекун
- Календарь приёмов: создание, статусы, виды по врачу
- Зубная карта: FDI взрослая + молочная + смешанный прикус, статусы зубов, выполненные процедуры с ценой
- Услуги/прайс
- Финансы (базово): счёт по визиту, оплата, долг пациента
- Настройки клиники и персонала
- i18n-инфраструктура, переводы **только AZ** (ключи готовы для RU/EN)

**Не входит в MVP:** склад, отчёты (кроме простой сводки на дашборде), SMS, документы-файлы, super-admin биллинг (тенанты создаются вручную).

## 8. Правила разработки

1. **`clinic_id` обязателен** в каждой бизнес-таблице и каждом запросе; изоляция — через middleware, не вручную в каждом эндпоинте.
2. **Ни одной hardcoded-строки в UI** — только i18n-ключи (`az.json` сейчас, `ru.json`/`en.json` позже). Формат дат/валюты — через локаль.
3. **Зубы — только нотация FDI**: постоянные `11–48`, молочные `51–85`. Поле `dentition: permanent | primary | mixed` у карты пациента; режим карты выбирается по возрасту автоматически, с ручным override.
4. **Детская логика — не отдельный модуль, а свойство пациента**: `patient.guardian_id`, возрастные проверки, согласия от опекуна, детский прайс — флаг на услуге.
5. **Soft delete** (`deleted_at`) для пациентов, визитов, счетов; жёсткое удаление — только Super Admin.
6. **Деньги** — integer (гяпики/копейки), валюта AZN в настройках тенанта; никаких float.
7. **Даты** — UTC в БД, таймзона клиники в настройках, отображение по локали.
8. **API**: server actions / route handlers с zod-валидацией; единый формат ошибок; права проверяются по роли на сервере, UI только скрывает.
9. **Миграции только через Prisma migrate**, схема — единственный источник правды.
10. **Аудит-лог** для изменений мед. и финансовых данных с v1.0.
11. **Без cross-tenant фич "на потом"** — любой новый код пишется в предположении изоляции.
12. **Каждая фича = страница + права по ролям + переводы + пустые состояния** — иначе не считается готовой.
13. **UI только по дизайн-системе** ([DESIGN.md](DESIGN.md)): сначала токены и базовые компоненты, страницы собираются из них; никаких inline-цветов и разнобоя иконок.
14. **Зубная карта — по спецификации** [DENTAL_CHART.md](DENTAL_CHART.md): единый enum статусов, tooth_records/procedures/history с tenant-изоляцией, компонент карты принимает набор зубов как данные (готовность к детскому режиму).
15. **Разработка маленькими итерациями:** анализ → план → реализация одного модуля; не переписывать работающее, не ломать существующую структуру.

### Рабочий процесс каждой задачи (обязательный)

**Перед изменениями:**
1. Проанализировать текущую структуру проекта (docs/ + код).
2. Кратко перечислить, какие файлы будут изменены и почему — до правок.
3. Делать только текущую задачу; никаких «заодно» — лишних модулей, рефакторингов, переименований таблиц/полей/компонентов/маршрутов без требования задачи.

**Во время:**
4. Не удалять рабочий код, не переписывать проект заново.
5. Каждая новая функция обязана поддерживать multi-tenant: `clinic_id`, `doctor_id`, проверка роли на сервере.
6. UI — строго по [DESIGN.md](DESIGN.md): navy фон, cyan/teal акценты, rounded cards, Lucide-иконки, responsive, без перегруза.

**После завершения — всегда отчёт:**
7. Список изменённых файлов · что добавлено · что проверить вручную · возможные риски/баги.
